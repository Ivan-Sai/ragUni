**Мета роботи** — розробити веб-систему семантичного пошуку відповідей в університетській документації на основі архітектури Retrieval-Augmented Generation з підтримкою рольового доступу, аутентифікації та видобування структурованих даних з графіків занять і протоколів сесії.

За результатами аналізу типових інформаційних потреб студентів, викладачів і адміністрації факультету розроблено повнофункціональну веб-систему, яка складається з backend-сервісу на FastAPI з векторним сховищем MongoDB Atlas Vector Search, frontend-додатку на Next.js, та конвеєра обробки документів, який підтримує формати PDF, DOCX, XLSX і TXT. Реалізовано детермінований парсер навчальних розкладів, який точно атрибутує заняття до груп через структурний аналіз таблиць PDF замість послідовності токенів LLM, що усуває типову проблему хибного зіставлення колонок при автоматичному вилученні. Впроваджено триетапну архітектуру пошуку: попередній аналіз запиту LLM з виділенням сутностей → мультистратегічне отримання кандидатів (структурний пошук + векторний пошук + повнотекстовий пошук) з фузією результатів за алгоритмом Reciprocal Rank Fusion → перерейтинг top-50 кандидатів LLM крос-енкодером з відсіканням нерелевантних результатів. Розроблено систему рольового доступу з трьома ролями (студент, викладач, адміністратор), яка реалізує жорстку фільтрацію видачі відповідно до факультету, групи, року й рівня навчання користувача.

**Методи:** ретрівл-аугментована генерація, контекстуальний пошук Anthropic, гібридний пошук з фузією рангів (Reciprocal Rank Fusion), векторні embedding'и multilingual-e5-large, LLM cross-encoder reranking, рольова модель доступу (RBAC), детерміноване структурне видобування таблиць.

**Інструменти:** Python 3.11, FastAPI, LangChain, MongoDB Atlas Vector Search, FastEmbed, Deepseek API, Next.js 15, React 19, TypeScript, Tailwind CSS, shadcn/ui, NextAuth.js, Pydantic v2, pdfplumber, pytest. # ВСТУП

**Обґрунтування вибору теми та її актуальність.** Університети накопичують значні обсяги документації — навчальні розклади, графіки сесій, методичні рекомендації, накази, силабуси дисциплін, положення про організацію освітнього процесу. Доступ до цієї інформації для здобувачів вищої освіти, викладачів і працівників адміністрації наразі реалізується переважно через статичні веб-сторінки факультетів, розсилки електронною поштою або особисті звернення до деканату. Такий підхід призводить до низки системних проблем: складність пошуку конкретної відповіді у багатосторінкових PDF-документах, відсутність персоналізації (студент бачить розклад усіх груп, навіть якщо потрібний лише розклад своєї), застарівання інформації між публікаціями, перевантаження адміністративного персоналу типовими запитаннями.

Розвиток великих мовних моделей (Large Language Models, LLM) і архітектури Retrieval-Augmented Generation (RAG) останніми роками створив принципову можливість побудови систем, які надають точні, обґрунтовані документами відповіді природною мовою. Класична робота N. Lewis та співавторів [1] заклала теоретичні основи RAG як підходу, що поєднує параметричну пам'ять мовної моделі з непараметричною пам'яттю векторного сховища. Подальші роботи (W. Wang та співавторів про embedding-моделі multilingual-e5 [2], рекомендації Anthropic щодо контекстуального пошуку [3], алгоритм фузії рангів G. Cormack та інших [4]) суттєво розширили арсенал методів і дозволили досягати високої точності відповідей навіть на корпусах документів, написаних українською мовою з домішками англомовних термінів.

Попри активний розвиток комерційних рішень загального призначення (OpenAI Assistants API, Anthropic Claude з підтримкою файлів, Google NotebookLM), потреба у спеціалізованому сервісі для українських вищих навчальних закладів залишається нерозв'язаною з кількох причин. По-перше, університетська документація має специфічну структуру (табличні розклади з вертикальним текстом, мультигрупові протоколи сесій, спеціалізовані формати документообігу), яка слабо обробляється вертикальними рішеннями. По-друге, доступ до окремих документів регулюється рівнем користувача — студент не повинен бачити внутрішніх наказів кафедри, а викладач не має отримувати персональні питання іншого викладача. По-третє, конфіденційність персональних даних здобувачів вищої освіти вимагає, щоб документація оброблялася на власній інфраструктурі університету, а не передавалася стороннім хмарним сервісам без відповідних угод.

**Зв'язок роботи з сучасними науковими і технічними знаннями.** Робота базується на сучасному стані розвитку обробки природної мови, інформаційного пошуку та системного програмування. Використано результати критичного аналізу сучасних публікацій з галузей: векторного семантичного пошуку (Karpukhin та ін. про Dense Passage Retrieval [5], Reimers і Gurevych про sentence-BERT [6]), оцінювання якості відповідей мовних моделей (Zheng та співавторів про LLM-as-a-judge [7]), архітектурних шаблонів сучасних RAG-систем (огляд Gao та ін. [8]), а також стандартів безпеки веб-сервісів (рекомендації OWASP щодо JWT, OAuth 2.0, CSP).

**Мета роботи** — розробити веб-сервіс, який забезпечує семантичний пошук відповідей в університетській документації з урахуванням ролі та аудиторії користувача, з підтримкою генерації відповідей природною мовою на основі вилучених релевантних фрагментів, та з можливостями адміністрування корпусу документів і користувачів.

**Методи дослідження.** Для досягнення поставленої мети використано: системний аналіз предметної області та аналіз існуючих рішень для виявлення вимог; теоретичні методи побудови архітектури веб-систем, конвеєрів обробки документів і ансамблів моделей пошуку; експериментальні методи оцінювання якості пошуку (метрики Recall@k, Precision@k, Mean Reciprocal Rank, Normalized Discounted Cumulative Gain); комп'ютерне моделювання роботи системи на наборі реальних документів факультету радіофізики, електроніки та комп'ютерних систем.

**Практичне значення отриманих результатів.** Розроблений програмний комплекс може застосовуватися як основа інтелектуальної довідкової служби факультету або університету. Архітектура передбачає горизонтальне масштабування (stateless backend, MongoDB Atlas з реплікацією), що дозволяє обслуговувати сотні користувачів одночасно. Проведено експериментальну апробацію на корпусі з семи документів навчальних розкладів і графіків сесій кафедри радіофізики, електроніки та комп'ютерних систем; результати наведено у Розділі 4.

**Особистий внесок здобувача вищої освіти.** Усі основні результати кваліфікаційної роботи отримано автором особисто. Зокрема, розроблено: архітектуру сервісу та структуру бази даних; детермінований парсер навчальних розкладів з підтримкою вертикального тексту й об'єднаних клітинок; триетапний конвеєр пошуку з аналізом запиту, мультистратегічним отриманням і перерейтингом; систему рольового доступу з трирівневою моделлю; frontend-додаток з повним циклом аутентифікації, чату та адміністрування; пакет автоматизованих тестів (256 тестів, що покривають критичні шляхи безпеки, бізнес-логіки та якості пошуку).

# 1 АНАЛІЗ ПРЕДМЕТНОЇ ОБЛАСТІ ТА ВИМОГИ ДО СИСТЕМИ

## 1.1 Особливості предметної області

Університет як об'єкт автоматизації характеризується високою різнорідністю інформаційних потоків. Документація, з якою працюють здобувачі вищої освіти, викладачі та адміністрація, поділяється на кілька типів за форматом, структурою та режимом доступу.

**Розклади навчальних занять** становлять найбільшу за обсягом і найскладнішу за структурою категорію документів. Типовий розклад факультету являє собою PDF-файл обсягом від трьох до шести сторінок, який містить таблицю з рядками за часовими слотами та колонками за академічними групами. Кількість груп у таблиці залежить від спеціальності й може досягати п'ятнадцяти, що призводить до використання вертикально орієнтованого тексту для назв днів тижня та значного звуження клітинок з назвами навчальних дисциплін. Окремо виділяються об'єднані клітинки, які охоплюють дві або більше суміжних колонок — такий формат позначає спільну лекцію для кількох груп. Сторінки-продовження часто не повторюють шапку таблиці, що ускладнює машинне зіставлення колонок з групами.

**Графіки залікової та екзаменаційної сесій** представлені у вигляді таблиць «дата — час — група — дисципліна — викладач — аудиторія». Графіки можуть охоплювати одночасно бакалаврів і магістрів усіх курсів, що робить документ одним із найбільших за обсягом серед усієї документації факультету. Інформація у графіку має чіткий часовий контекст, що дозволяє відповідати на запити типу «коли в нашої групи іспит з фізики?».

**Методичні рекомендації, положення та накази** є прозовою документацією, побудованою за ієрархією: розділ → стаття → пункт → підпункт. Пошук у такій документації потребує точного цитування з посиланням на номер пункту, оскільки користувач часто шукає конкретну норму, а не загальний опис теми.

**Силабуси та робочі програми навчальних дисциплін** структуровані за схемою «дисципліна → змістовий модуль → тема → програмні результати навчання». Цей тип документа близький за форматом до методичних рекомендацій, але має додаткову прив'язку до кодів спеціальності та форм атестації.

**Об'яви та інформаційні листи** є короткими документами з низькою повторюваністю структури, які найкраще обробляються універсальним рекурсивним розбиттям тексту на семантичні сегменти.

З точки зору режиму доступу, документація поділяється на:

- **публічну** — доступну будь-кому без аутентифікації (факультетські оголошення, контакти);
- **факультетську** — доступну зареєстрованим користувачам відповідного факультету (накази, методичні рекомендації);
- **обмежену** — доступну лише викладачам і адміністраторам (внутрішні протоколи, фінансова інформація).

Окремий рівень фільтрації — **аудиторія документа**: розклад певної групи має бути недоступним для пошуку студента з іншої групи, навіть якщо обидва належать до одного факультету. Цей рівень фільтрації не може бути реалізований класичною моделлю «папка-група» через перекриття аудиторій (наприклад, спільна лекція для двох груп) і вимагає спеціального механізму на рівні окремих фрагментів документа.

## 1.2 Аналіз існуючих аналогів

Аналіз сучасного ринку показав відсутність комплексного рішення, яке б одночасно враховувало специфіку української академічної документації, рольову модель доступу та інтеграцію з адміністративними процесами факультету. Розглянуто чотири класи систем-аналогів.

**Корпоративні платформи семантичного пошуку загального призначення** (Glean, Coveo, Elastic Enterprise Search) реалізують повноцінний RAG-конвеєр для документів організації, мають розвинений аналітичний інтерфейс і моделі доступу на рівні документа. Перевагами є зрілість, висока якість пошуку для англомовного контенту та масштабованість. Серед недоліків — висока вартість підписки (від кількох тисяч доларів США щомісяця), орієнтація на корпоративні джерела даних (Confluence, SharePoint, Slack), яких немає в типовому університеті, відсутність спеціалізованої обробки табличних розкладів і слабка підтримка української мови.

**Універсальні чат-боти з підтримкою документів** (OpenAI ChatGPT з функцією «Files», Anthropic Claude з вкладенням файлів, Google NotebookLM) дозволяють користувачу самостійно завантажити документ і задавати питання за його змістом. Перевагою є простота використання та якість LLM. Принципові обмеження для застосування у ВНЗ: відсутність спільного корпусу, кожен користувач повинен завантажувати документи самостійно; передача документів стороннім сервісам, що несумісно з вимогами законодавства про захист персональних даних здобувачів; відсутність моделі ролей та аудиторії, неможливість гарантувати, що відповідь побудовано лише на дозволених для користувача документах.

**Спеціалізовані рішення для освіти** (Microsoft Copilot for Education, IBM Watson Education) є повноцінними платформами підтримки навчального процесу з RAG-функціональністю. Основні недоліки в українському контексті — прив'язка до екосистеми постачальника (Microsoft 365 або IBM Cloud), необхідність значних інфраструктурних інвестицій, відсутність безкоштовних варіантів для малих факультетів і слабка українська локалізація.

**Чат-боти, побудовані на правилах** (rule-based bots), які використовуються в деяких українських ВНЗ для відповідей на типові питання, мають простоту впровадження як єдину перевагу. Недоліки очевидні: вузький набір підтримуваних запитів, обмежений шаблонами розробника; нездатність працювати з вільними формулюваннями; відсутність генеративного компоненту, що змушує бота відповідати посиланнями на сторінку сайту замість витягу конкретного факту.

## 1.3 Загальна задача роботи

На основі аналізу предметної області та існуючих аналогів сформульовано загальну задачу роботи: спроектувати, реалізувати та провести експериментальну апробацію веб-сервісу інтелектуального пошуку, який забезпечує:

1. індексацію документів університету у форматах PDF, DOCX, XLSX, TXT з автоматичною класифікацією за типом документа та видобуванням структурованих даних з розкладів і протоколів сесії;
2. семантичний пошук відповідей природною мовою (українська, англійська) з обґрунтуванням посиланнями на конкретні фрагменти джерел;
3. рольовий доступ з трьома рівнями (студент, викладач, адміністратор) і двома шарами фільтрації — за рівнем доступу документа та за аудиторією окремого фрагмента;
4. адміністрування корпусу документів і користувачів через веб-інтерфейс;
5. вимірювані характеристики якості пошуку (Recall@k, Precision@k, MRR, nDCG@k) з можливістю автоматичного відстеження регресії після зміни моделі чи алгоритму.

## 1.4 Вимоги до програмного засобу

### 1.4.1 Функціональні вимоги

Програмний засіб повинен реалізовувати такі функції:

- **F1. Аутентифікація користувачів** — реєстрація з підтвердженням адміністратора, вхід за допомогою пари «адреса електронної пошти/пароль», видача пари токенів JWT (access + refresh), оновлення access-токена за refresh-токеном, вихід зі скасуванням refresh-токена, відновлення пароля через електронну пошту.
- **F2. Управління профілем** — самостійне редагування неструктурованих полів профілю користувача; редагування структурованих полів (факультет, група, рік, рівень навчання) — лише адміністратором.
- **F3. Адміністрування користувачів** — перегляд списку, фільтрація за статусом, схвалення/відхилення заявок викладачів, блокування/розблокування облікових записів, зміна ролі.
- **F4. Управління довідником факультетів і груп** — створення, редагування, перегляд, видалення з перевіркою посилальної цілісності.
- **F5. Завантаження документів** — підтримка форматів PDF/DOCX/XLSX/TXT обсягом до 10 МБ, автоматична класифікація за типом, вибір рівня доступу та цільової аудиторії (групи, роки, рівень навчання).
- **F6. Перегляд і видалення документів** — список з фільтрацією, попередній перегляд тексту, видалення з каскадним очищенням індексу.
- **F7. Семантичний пошук відповідей** — приймання питання природною мовою, повернення відповіді з посиланнями на джерела, врахування історії діалогу в межах сесії.
- **F8. Чат-історія** — збереження сесій діалогу за користувачем, перегляд і видалення сесій.
- **F9. Зворотний зв'язок** — оцінка відповіді (схвально / несхвально) для подальшого аналізу.
- **F10. Аудит** — фіксація привілейованих дій (підтвердження користувача, видалення документа, зміна ролі) для подальшої перевірки.

### 1.4.2 Нефункціональні вимоги

- **N1. Якість пошуку** — на еталонному наборі з мінімум 12 питань різних типів забезпечити Recall@5 ≥ 0,28, MRR ≥ 0,49, nDCG@5 ≥ 0,31. Відстеження регресії — автоматичне через CI.
- **N2. Час відгуку** — медіана часу відповіді на запит до пошуку ≤ 5 секунд (без урахування часу LLM-генерації); час відповіді на адміністративні запити ≤ 500 мс.
- **N3. Доступність** — добова доступність сервісу не нижче 99 %.
- **N4. Масштабованість** — підтримка одночасної роботи не менше 100 користувачів на стандартній інстансі без деградації часу відгуку.
- **N5. Безпека** — відповідність типовим вимогам OWASP Top 10 для веб-додатків; відсутність відомих критичних уразливостей у бібліотеках, що використовуються.

### 1.4.3 Вимоги до інтерфейсу користувача та платформи

Веб-інтерфейс повинен бути сумісним з останніми версіями браузерів Chrome, Firefox, Safari, Edge; адаптуватися до екранів від 320 пкс (мобільні) до 1920 пкс (стаціонарні); підтримувати темну тему; реалізовувати інтернаціоналізацію українською та англійською мовами без жорстко прописаних рядків у коді компонентів. Архітектура backend-сервісу повинна забезпечувати горизонтальне масштабування шляхом запуску кількох ідентичних копій за load balancer'ом.

### 1.4.4 Вимоги до безпеки та контролю доступу

Доступ до даних повинен регулюватися двома незалежними шарами:

- шар автентифікації — підпис JWT-токенів з обов'язковою перевіркою алгоритму, видавця та аудиторії; неможливість приймання неподписаних токенів навіть за умови помилкової конфігурації; revocation refresh-токенів через серверний allowlist; блокування облікового запису після п'яти невдалих спроб входу за п'ятнадцять хвилин;
- шар авторизації — фільтрація видачі пошуку за роллю користувача та аудиторією окремих фрагментів; неможливість читання документів іншого факультету для студентів та викладачів; перевірка володіння ресурсом для дій видалення / редагування.

Зберігання паролів у базі даних повинно реалізовуватися виключно у формі bcrypt-хешу; токени відновлення пароля — у формі SHA-256 хешу, що унеможливлює використання токена за умови читання бази. Усі привілейовані дії повинні фіксуватися в журналі аудиту з обмеженою TTL-політикою.

## 1.5 Завдання, які мають бути вирішені у кваліфікаційній роботі

Виходячи із загальної задачі роботи (підрозділ 1.3) та сформульованих вимог (підрозділ 1.4), визначено такі конкретні завдання, що мають бути вирішені у наступних розділах:

1. **Провести аналіз** сучасних архітектурних шаблонів систем семантичного пошуку та обґрунтувати вибір базової архітектури Retrieval-Augmented Generation з урахуванням специфіки університетської документації.
2. **Розробити** загальну архітектуру програмного засобу з виділенням клієнтського додатку, серверної частини, шару зберігання даних, конвеєра обробки документів і конвеєра пошуку відповідей.
3. **Спроектувати** структуру бази даних з підтримкою реляційних зв'язків між довідниками факультетів і груп, рольової моделі користувачів, документів з метаданими аудиторії та індексованих фрагментів зі вбудовуваннями.
4. **Розробити** конвеєр обробки документів, який поєднує детермінований розбір табличних структур (для розкладів і протоколів сесії) з універсальним розбиттям прозового контенту на семантичні фрагменти.
5. **Реалізувати** триетапний конвеєр пошуку: аналіз запиту LLM з виділенням сутностей, мультистратегічне отримання кандидатів з фузією рангів за алгоритмом Reciprocal Rank Fusion, перерейтинг top-50 кандидатів LLM крос-енкодером.
6. **Обґрунтувати** вибір моделі вбудовувань multilingual-e5-large для роботи з україномовною документацією та розробити обгортку, що автоматично додає префікси `query:` / `passage:`, передбачені авторами моделі.
7. **Реалізувати** систему рольового доступу з трьома рівнями (студент, викладач, адміністратор) та двома шарами фільтрації (рівень доступу документа + аудиторія окремого фрагмента) на основі MongoDB Atlas Vector Search pre-filter.
8. **Розробити** комплекс заходів безпеки веб-сервісу: пінінг алгоритмів JWT, серверний allowlist refresh-токенів, блокування облікових записів після серії невдалих спроб входу, нонс-Content Security Policy для frontend-додатку, захист від типових атак на завантаження документів (підміна MIME-типу, ZIP-bomb, обхід шляху).
9. **Спроектувати та реалізувати** клієнтський веб-додаток на Next.js з повним циклом аутентифікації, інтерфейсом чату з потоковим відображенням відповіді, адміністративними сторінками для управління корпусом і користувачами, інтернаціоналізацією українською та англійською мовами.
10. **Розробити** експериментальний набір даних з реальних документів кафедри для оцінки якості пошуку та реалізувати автоматизовану перевірку регресії метрик пошуку (Recall@k, Precision@k, MRR, nDCG@k) з прив'язкою до контрольних значень.
11. **Провести** експериментальну апробацію розробленого сервісу на реальному корпусі документів, проаналізувати отримані метрики, виявити сильні та слабкі сторони обраних архітектурних рішень.
12. **Покрити** критичні шляхи системи (аутентифікація, рольовий доступ, конвеєр пошуку, обробка документів) автоматизованими тестами зі стабільним прогоном у середовищі неперервної інтеграції.

# 2 МЕТОДИКА ПРОЕКТУВАННЯ ТА АРХІТЕКТУРА СИСТЕМИ

## 2.1 Загальна архітектура програмного засобу

На основі проведеного у Розділі 1 аналізу обрано трирівневу архітектуру з виділенням клієнтського додатку, серверної частини та шару зберігання даних. Загальну схему системи наведено на рисунку 2.1.

*[Рисунок 2.1 — Загальна архітектура веб-системи RAG-пошуку: клієнт (Next.js), сервер (FastAPI), сховище (MongoDB Atlas), зовнішній сервіс LLM (Deepseek API)]*

Рисунок 2.1 — Загальна архітектура веб-системи RAG-пошуку

Архітектура спроектована за принципом stateless backend: серверний додаток не зберігає стан користувача в оперативній пам'яті, всі сесії автентифікації представлені JWT-токенами, що дозволяє запускати кілька ідентичних копій сервера за load balancer'ом для горизонтального масштабування. Стан системи зберігається у MongoDB Atlas, який сам забезпечує реплікацію та відмовостійкість на рівні інфраструктури.

Клієнтський додаток повністю відокремлений від сервера й комунікує через REST API з підтримкою серверних подій (Server-Sent Events, SSE) для потокової передачі відповідей чату. Така відокремленість дозволяє використовувати один і той самий сервер з кількома різними клієнтами (web, mobile, telegram-bot) без модифікації серверного коду.

Усередині backend-сервісу архітектура побудована за шаблоном «шари» (layered architecture):

1. **Шар API** (`app/api/v1/`) — обробка HTTP-запитів, валідація вхідних даних через Pydantic, серіалізація відповідей. Цей шар не містить бізнес-логіки.
2. **Шар сервісів** (`app/services/`) — бізнес-логіка: парсинг документів, конвеєр RAG, рольовий доступ, аудит. Сервіси не залежать від HTTP-фреймворку, що спрощує тестування.
3. **Шар даних** (`app/models/`) — Pydantic-моделі та обгортки навколо MongoDB-колекцій.
4. **Шар інфраструктури** (`app/core/`) — крос-функціональні компоненти: безпека, обмеження частоти, логування, обробка помилок, спостережуваність.

## 2.2 Обґрунтування вибору технологічного стеку

### 2.2.1 Backend-фреймворк

Для серверної частини обрано **FastAPI** — Python-фреймворк, побудований на стандарті ASGI (Asynchronous Server Gateway Interface) з автоматичною генерацією OpenAPI-специфікації. Розглянуто три альтернативи (таблиця 2.1).

Таблиця 2.1 — Порівняння Python-фреймворків для backend

| Критерій | Flask | Django REST Framework | FastAPI |
| --- | --- | --- | --- |
| Підтримка асинхронного коду | через розширення | обмежена | нативна (async/await) |
| Автоматична OpenAPI-документація | ні | через розширення | вбудована |
| Валідація даних через типи | ні | через серіалізатори | через Pydantic |
| Залежність від ORM | ні | сильна (Django ORM) | відсутня |
| Швидкість обробки запитів | середня | низька | висока |

Підтримка асинхронного коду критична для роботи з MongoDB через драйвер Motor та для виклику зовнішніх API LLM, де очікування відповіді мережевого сервісу займає основну частину часу обробки. Pydantic-валідація на основі type hints дозволяє описувати схеми запитів і відповідей як звичайні класи Python, з автоматичною серіалізацією, валідацією та документацією без дублювання визначень.

### 2.2.2 База даних і векторне сховище

Для зберігання даних обрано **MongoDB Atlas** з підтримкою Vector Search. Альтернативи розглянуті у таблиці 2.2.

Таблиця 2.2 — Порівняння варіантів векторного сховища

| Критерій | PostgreSQL + pgvector | Pinecone | Weaviate | MongoDB Atlas Vector Search |
| --- | --- | --- | --- | --- |
| Сховище для основних даних | реляційне (потребує окремої моделі для документів) | відсутнє | у комплекті | документо-орієнтоване (зручне для PDF-метаданих) |
| Векторний пошук | до 1М векторів стабільно | необмежено (керована послуга) | необмежено (з кластеризацією) | до десятків мільйонів |
| Гібридний пошук (vector + full-text) | через окремі індекси | вбудований | вбудований | вбудований (Atlas Search + Vector Search) |
| Управління інфраструктурою | повністю на користувача | повністю керована | напівкерована | повністю керована |
| Pre-filter на рівні індексу | через WHERE | вбудований | вбудований | вбудований |
| Тарифікація | вартість сервера | за запити | за вузли | за обсяг кластера |

Ключовий критерій для розробленого сервісу — **pre-filter на рівні індексу**, тобто можливість фільтрувати кандидатів векторного пошуку за метаданими (рівень доступу, факультет, аудиторія) ще до обчислення відстані. Альтернативний підхід (post-filter) призводить до значного зниження ефективної глибини пошуку: якщо заявлено top-k=5 і відсівається 80 % через post-filter, фактично доступний лише top-1, що руйнує якість retrieval. Pre-filter забезпечує точність top-k у видимому користувачу результаті.

### 2.2.3 Модель вбудовувань

Для побудови векторного представлення документів обрано **multilingual-e5-large** (intfloat/multilingual-e5-large) — sentence-transformer-модель з архітектурою на основі XLM-RoBERTa, навчена на множині паралельних корпусів понад 100 мовами. Розглянуто три альтернативи (таблиця 2.3).

Ключова особливість моделі multilingual-e5 — обов'язкове використання префіксів `query:` для пошукових запитів та `passage:` для індексованих документів. Без префіксів якість пошуку падає на ~15 % від оптимальної; вимірювання на еталонному наборі підтвердило цю цифру. Оскільки префікси повинні застосовуватися автоматично у двох різних точках конвеєра (індексація та пошук), розроблено обгортку `_E5PrefixedEmbeddings`, яка успадковує від `FastEmbedEmbeddings` і додає відповідний префікс у методах `embed_documents` та `embed_query`.

### 2.2.4 Велика мовна модель

Для генерації відповідей та допоміжних задач (аналіз запиту, перерейтинг, видобування структурованих даних) обрано **Deepseek API** — китайський сервіс з OpenAI-сумісним інтерфейсом. Альтернативи: OpenAI GPT-4o-mini, Anthropic Claude Sonnet, Google Gemini Pro. Рішення прийнято за такими аргументами: (а) вартість токену на порядок менша за провідні західні сервіси, що критично для університетського застосування з обмеженим бюджетом; (б) OpenAI-сумісний інтерфейс дозволяє замінити сервіс на будь-який інший без переписування коду — достатньо змінити base URL і ключ; (в) висока якість обробки україномовних запитів, перевірена експериментально на тестовому корпусі.

### 2.2.5 Frontend-фреймворк

Для клієнтського додатку обрано **Next.js 15** з підтримкою React Server Components. Основні чинники вибору: server-side rendering для SEO та швидкого першого відображення; автоматичний routing з підтримкою динамічних сегментів (`/chat/[sessionId]`); вбудована підтримка edge middleware (використано для аутентифікації на рівні сервера ще до рендеру сторінки); зрілість екосистеми React-компонентів (UI-кит shadcn/ui, що базується на Radix UI primitives); підтримка інтернаціоналізації через `next-intl`.

Альтернативи (Vue.js + Nuxt 3, SvelteKit, Angular) розглянуто, але не обрано через меншу зрілість екосистеми та відсутність команди-розробника з відповідною компетенцією.

## 2.3 Структура бази даних

База даних розроблена на основі документо-орієнтованої моделі MongoDB. Схема складається з восьми основних колекцій. ER-діаграма наведена на рисунку 2.2.

*[Рисунок 2.2 — ER-діаграма бази даних: users, faculties, groups, documents, document_chunks, chat_history, audit_logs, refresh_tokens, feedback]*

Рисунок 2.2 — ER-діаграма бази даних

**users** — облікові записи користувачів. Поля: `_id`, `email` (унікальний), `hashed_password` (bcrypt), `full_name`, `role` (`student` / `teacher` / `admin`), `faculty_id`, `group_id`, `year`, `level` (`bachelor` / `master` / `phd`), `department`, `position`, `is_approved`, `is_active`, `failed_login_attempts`, `last_failed_login`, `locked_until`, `password_reset_token_hash`, `created_at`, `updated_at`. Індекси: унікальний на `email`, складені на `faculty_id` та `group_id` для швидкого видалення з довідника.

**faculties** — довідник факультетів. Поля: `_id`, `name`, `name_lower` (для регістронезалежної унікальності), `description`. Унікальний індекс на `name_lower`.

**groups** — довідник академічних груп. Поля: `_id`, `name`, `name_lower`, `faculty_id` (зв'язок з `faculties`), `level`. Унікальний складений індекс на `(faculty_id, name_lower)`, що гарантує відсутність двох груп з однаковою назвою в межах факультету при дозволі однойменних груп у різних факультетах.

**documents** — записи про завантажені документи. Поля: `_id`, `filename`, `file_type`, `access_level` (`public` / `faculty` / `restricted`), `faculty_id`, `target_group_ids`, `target_years`, `target_level`, `uploaded_at`, `uploaded_by_id`, `total_chunks`, `extracted_text`, `extraction_method`, `structured_records`, `structured_records_count`, `doc_type`. Поле `structured_records` зберігає список структурованих записів для розкладів і протоколів сесії, що дозволяє виконувати точний пошук за конкретними значеннями (день, час, дисципліна) без векторної семантики.

**document_chunks** — індексовані фрагменти документів зі вбудовуваннями. Поля: `_id`, `text`, `embedding` (масив 1024 чисел з рухомою комою), `source_file`, `document_id`, `file_type`, `access_level`, `faculty_id`, `target_group_ids`, `target_years`, `target_level`, `group_label`, `year_label`, `level_label`, `record_type`, `doc_type`, `chunk_index`, `total_chunks`, `chunk_length`. Індекси Atlas Vector Search: knnVector на `embedding` з фільтрами на `access_level`, `faculty_id`, `target_group_ids`, `target_years`, `target_level`, `doc_type`. Всі поля фільтрації включено до конфігурації індексу — без цього pre-filter не працює.

**chat_history** — сесії чату. Поля: `_id`, `session_id`, `user_id`, `messages` (масив повідомлень з `role` / `content` / `timestamp` / `sources`), `title`, `created_at`, `updated_at`. Складений унікальний індекс на `(session_id, user_id)` запобігає колізіям між сесіями різних користувачів, які можуть випадково отримати однаковий ідентифікатор.

**audit_logs** — журнал привілейованих дій. Поля: `_id`, `timestamp`, `actor_id`, `actor_role`, `action` (`user.approved`, `user.blocked`, `user.role_changed`, `user.password_changed`, `document.uploaded`, `document.deleted`, `chat.session_deleted`), `resource_type`, `resource_id`, `metadata`. TTL-індекс на `timestamp` з терміном 365 днів автоматично очищає застарілі записи.

**refresh_tokens** — серверний allowlist refresh-токенів. Поля: `_id`, `jti` (унікальний), `user_id`, `token_hash` (SHA-256), `expires_at`, `revoked`, `created_at`, `revoked_at`. TTL-індекс на `expires_at` автоматично видаляє прострочені записи. Allowlist дозволяє відкликати конкретний токен (logout) або всі токени користувача (зміна пароля), що неможливо у класичній stateless JWT-схемі.

**feedback** — оцінки відповідей чату. Поля: `_id`, `user_id`, `session_id`, `message_index`, `rating` (`thumbs_up` / `thumbs_down`), `comment`, `created_at`. Унікальний складений індекс на `(user_id, session_id, message_index)` гарантує, що користувач має одну оцінку на повідомлення (повторна оцінка перезаписує попередню).

## 2.4 Конвеєр обробки документів

Конвеєр приймає завантажений файл та перетворює його на сукупність індексованих фрагментів. Послідовність обробки наведена на рисунку 2.3.

*[Рисунок 2.3 — Конвеєр обробки документа: парсинг файлу → класифікація → побудова контексту документа → видобування за типом → розбиття на фрагменти → embedding → запис у MongoDB]*

Рисунок 2.3 — Конвеєр обробки документа

**Етап 1. Валідація та парсинг.** Файл проходить через перевірку розширення, зіставлення з MIME-типом (через бібліотеку python-magic для виявлення підміни розширення), обмеження максимального розміру (10 МБ), та для форматів OOXML (DOCX, XLSX) — перевірку, що сукупний обсяг розпакованих компонентів не перевищує 100 МБ (захист від zip-bomb). Парсинг PDF реалізовано на основі pdfplumber, який забезпечує доступ як до прозового тексту, так і до табличних структур з координатами клітинок. DOCX парситься через python-docx, XLSX — через pandas з ядром openpyxl.

**Етап 2. Класифікація типу документа.** Витягнутий текст подається на LLM-класифікатор, що повертає одну з міток: `schedule` (навчальний розклад), `exam_protocol` (графік сесії), `regulation` (положення/наказ), `curriculum` (силабус/робоча програма), `prose` (загальна прозова документація), `tabular` (табличний документ невідомого типу), `unknown`. Класифікація виконується одним коротким викликом моделі (~150 токенів) і кешується на рівні документа.

**Етап 3. Контекстуалізація.** За методом Anthropic Contextual Retrieval (опубліковано у [3]) для кожного документа формується одно-двореченнєвий опис контексту, що буде додаватися до кожного фрагмента перед обчисленням вектора. Підхід підвищує точність пошуку приблизно на 35 %, оскільки фрагмент середини документа без контексту може втратити прив'язку до загальної теми (наприклад, фрагмент з назвою предмета без вказівки факультету).

**Етап 4. Тип-специфічне видобування.** Залежно від класифікації використовується відповідний extractor:

- *Schedule extractor* — гібридна стратегія: спочатку працює детермінований парсер (підрозділ 2.5), потім LLM-уточнення кожної клітинки (асинхронно, батчами по 25 клітинок, паралельно через `asyncio.gather`). Виходом є список структурованих записів типу `{day, time, group, subject, teacher, room, lesson_kind}`. Кожен запис стає окремим фрагментом для індексації.
- *Regulation / Curriculum extractor* — розбиття за заголовками: «Стаття N», «Розділ N», «Пункт N» для регламентної документації; «Змістовий модуль N», «Тема N», «Силабус» для силабусів. Кожен розділ стає одним фрагментом; розділи понад 1500 символів додатково розбиваються на параграфи.
- *Prose / unknown extractor* — рекурсивне розбиття на семантичні фрагменти з пріоритетом меж: подвійний переніс рядка → переніс → крапка → пробіл. Розмір фрагмента — 1000 символів з перекриттям 200, що відповідає типовим рекомендаціям для RAG-систем.

**Етап 5. Embedding.** Кожен фрагмент проходить через E5-обгортку, яка додає префікс `passage:` перед обчисленням вектора. Embeddings обчислюються локально через FastEmbed (квантизований ONNX-інференс), пакетами по 100 фрагментів.

**Етап 6. Збереження.** Фрагменти зі вбудовуваннями та метаданими записуються в колекцію `document_chunks`. Метадані аудиторії (`target_group_ids`, `target_years`, `target_level`) копіюються з рівня документа на рівень кожного фрагмента — це дозволяє Atlas Vector Search фільтрувати на рівні pre-filter без додаткового JOIN.

## 2.5 Детермінований парсер навчальних розкладів

Класичний підхід до видобування даних з табличних PDF — подача markdown-представлення таблиці на LLM, з проханням повернути JSON. На практиці цей підхід має критичну ваду: LLM рахує позицію колонки за послідовністю символів-розділювачів `|`, що в таблиці з 12 колонками призводить до ~30 % помилок зіставлення. Експериментально на тестовому розкладі з 91 заняттям отримано 28 неправильно атрибутованих груп з 91. Спостережувана картина: запис, що насправді належить групі КСМ-1 магістри, потрапляє у видачу як заняття групи СА-2 бакалаври, що для користувача є серйозною помилкою.

Розроблений детермінований парсер усуває проблему атрибуції колонок шляхом прямої роботи зі структурою таблиці на рівні pdfplumber. Послідовність кроків:

1. **Виявлення часової колонки.** Серед лівих колонок (індекси 0..3) знаходиться та, що містить найбільше клітинок, які відповідають регулярному виразу часового слоту (`\d{1,2}[:.]?\d{2}\s*[-–—]\s*\d{1,2}[:.]?\d{2}`).
2. **Виявлення колонки днів тижня.** Серед колонок ліворуч від часової шукається та, в якій найбільше клітинок розпізнається як назва дня тижня (з використанням словника варіантів: `понеділок`, `вівторок`, `середа` тощо).
3. **Виявлення шапки таблиці.** Над першим рядком даних знаходиться сегмент шапки. Останній рядок шапки, в якому права частина рядка містить переважно короткі коди груп, інтерпретується як рядок назв груп. Над ним — рядок з пари «N курс / бакалавр|магістр», який пере-проектується на колонки за допомогою forward-fill (об'єднана клітинка).
4. **Побудова відображення колонка→група.** Для кожної колонки правіше часової формується запис `ColumnSpec(index, group, year, level)`.
5. **Розбір даних.** Послідовно обробляються рядки даних; рядок, що містить часовий слот, є «якорем» нового заняття; решта рядків доповнюють активний слот. Кожне заповнення клітинки розв'язується у запис `CellEvent(day, time, group, year, level, raw_text)`.

Окрему складність створюють вертикально написані назви днів, які pdfplumber повертає як послідовність окремих клітинок, кожна з одним символом. Розроблений алгоритм destacking реконструює оригінальне слово у дві проходки: (а) within-cell — клітинка з пробілами між літерами одного слова збирається назад в одне слово; (б) across-row — послідовність клітинок з одним символом з'єднується у вертикальний стек. Алгоритм підтримує обидві орієнтації написання (top-down і bottom-up) та коректно обробляє апострофи (`п'ятниця`).

Сторінки-продовження розкладу часто не повторюють шапку, що унеможливлює детектування колонок класичним способом. Реалізовано перенос канонічного вертикального грид-у з першої сторінки на всі наступні через зіставлення x-координат клітинок. Об'єднані клітинки, що візуально перетинають кілька канонічних колонок, розпізнаються за x-bbox і реплікуються у всі перетинаючі колонки — це коректно відтворює спільні лекції для двох груп.

Після детермінованого розбору отримані сирі тексти клітинок пропускаються через LLM-уточнювач, який розкладає рядок типу «Бойко Ю.В. Інтерфейси систем обміну даними (лек.) ауд.45» на структуровані поля `{teacher: "Бойко Ю.В.", subject: "Інтерфейси систем обміну даними", lesson_kind: "лек.", room: "ауд.45"}`. Уточнення виконується батчами по 25 клітинок паралельно через `asyncio.gather`, що утримує загальний час обробки документа в межах 30 секунд для типового розкладу з 90 клітинками.

## 2.6 Конвеєр пошуку відповідей (RAG)

Розроблений конвеєр пошуку є триетапним; його загальна схема наведена на рисунку 2.4.

*[Рисунок 2.4 — Триетапний конвеєр пошуку: query analyzer → multi-strategy retrieval (structured + vector + hybrid) → RRF fusion → LLM reranker → context assembly → answer generation]*

Рисунок 2.4 — Триетапний конвеєр пошуку відповідей

**Етап 1. Аналіз запиту.** Запит користувача та коротка історія діалогу подаються на спеціалізований LLM-аналізатор з нульовою температурою. На виході — структура `QueryAnalysis`, що містить: класифікований intent (`schedule_lookup`, `exam_dates`, `person_lookup`, `concept_explanation`, `document_search`, `general`), флаг `is_personal` (питання стосується власних даних користувача), словник виділених сутностей (`day_of_week`, `date_range`, `subject`, `teacher`, `room`), переформульований запит з розв'язанням займенників, перелік пріоритетних стратегій пошуку (`structured`, `vector`, `lexical`) і список цільових типів документів. У разі недоступності LLM (timeout 12 секунд або помилка JSON-парсингу) спрацьовує fallback-аналізатор на регулярних виразах, який виділяє день тижня та intent з обмеженого словника.

**Етап 2. Мультистратегічне отримання.** За результатами аналізу запускаються одна або кілька стратегій пошуку, що працюють паралельно через `asyncio.gather`:

- *Structured retrieval* — точний фільтр над `documents.structured_records` за полями аудиторії (група, рік, рівень) і виділеними сутностями (день, дисципліна). Високопріоритетний шлях для розкладних запитів типу «що у мене у вівторок?»; повертає до 80 кандидатів.
- *Vector retrieval* — кореневий пошук у MongoDB Atlas Vector Search з pre-filter за рівнем доступу та аудиторією; повертає top-K кандидатів з фільтрацією за поріг косинусної схожості (0,65). Поріг застосовується до фузії — це уникає потрапляння у контекст шуму, який далі змусить LLM придумувати неіснуючі факти.
- *Hybrid retrieval* — гібридний пошук Atlas (vector + full-text BM25) з reciprocal rank fusion на стороні Atlas. Активується для запитів з виділеним конкретним терміном.

**Етап 3. Фузія та перерейтинг.** Результати трьох стратегій об'єднуються алгоритмом Reciprocal Rank Fusion з параметром `k=60` (рекомендовано у [4]). Формула RRF: `score(d) = Σ 1 / (k + rank_i(d))` за всіма стратегіями, в яких документ d з'явився. Документи, що з'являються у кількох стратегіях, отримують підвищений ранг.

Топ-50 кандидатів передається на LLM-крос-енкодер, який оцінює релевантність кожного кандидата запиту за шкалою 0..10 одним викликом моделі. Кандидати з оцінкою менше 1,0 відсіюються; решта впорядковується за оцінкою; до фінального контексту потрапляє top-15 (або менше, залежно від налаштувань `top_k_results`).

**Етап 4. Генерація відповіді.** Відсортовані фрагменти форматуються у єдиний контекст з нумерованими маркерами `[Джерело N]`. Сформований контекст разом із системним промптом і запитом подається на головну модель Deepseek. Системний промпт містить інструкції щодо мови відповіді (відповідати мовою запиту), вимог до повноти (перерахувати усі знайдені записи без пропусків) і обов'язкових цитат за форматом `[N]`.

Відповідь стрімається назад клієнту через Server-Sent Events, щоб користувач починав бачити текст відразу після генерації перших токенів, не очікуючи завершення всієї відповіді.

Якщо найкращий кандидат після перерейтингу має оцінку нижчу за поріг no-answer (0,55), система видає захищену від галюцинації відповідь типу «Не можу знайти відповідь на це питання у наявних документах» замість того, щоб спробувати побудувати відповідь на слабких джерелах. Цей захист критичний для академічного контексту, де хибна відповідь з виглядом на надійну може ввести користувача в оману.

## 3.2 Реалізація аутентифікації та авторизації

### 3.2.1 JWT з пінінгом алгоритму та обов'язковими claims

Токени видаються з обов'язковими claims `iss` (issuer), `aud` (audience), `iat` (issued at), `exp` (expiration), `jti` (JWT ID), `sub` (subject — email користувача), `type` (`access` або `refresh`). Декодер відмовляє у прийнятті токена, якщо: алгоритм підпису не входить до обмеженого списку (`HS256`/`HS384`/`HS512`); відсутній будь-який обов'язковий claim; значення `iss` або `aud` не співпадає з налаштованими для сервісу. Окрім фактичного декодування пінінг алгоритму перевіряється і на рівні модуля (assertion при імпорті) — навіть якщо хтось у майбутньому пом'якшить валідатор у налаштуваннях, декодер відмовиться приймати токени з небезпечним алгоритмом.

Особлива увага приділена захисту від атаки «alg: none», коли зловмисник підставляє у токен заголовок з вказівкою алгоритму `none` (не підписаний токен). Експериментально перевірено, що декодер відмовляє у прийнятті такого токена з винятком `JWTError` (відповідний автоматизований тест включено до `tests/unit/test_jwt_security.py`).

### 3.2.2 Рефреш-токени з серверним allowlist

Класична stateless-схема JWT не дозволяє відкликати конкретний токен — токен валідний до закінчення терміну `exp`, навіть якщо користувач змінив пароль або вийшов з системи. У розробленому сервісі цю проблему розв'язано через серверний allowlist refresh-токенів (`services/refresh_tokens.py`).

При видачі refresh-токена в колекцію `refresh_tokens` записується кортеж `(jti, user_id, token_hash, expires_at, revoked=False)`. Token hash — SHA-256 від сирого токена; це дозволяє: перевірити, що пред'явлений токен є саме тим, що видавався (захист від колізій jti); не зберігати у БД сирий токен (захист від витоку через читання БД).

Endpoint `POST /auth/refresh` спочатку декодує JWT, потім перевіряє, що відповідний `jti` присутній у allowlist і не позначений як `revoked`, і що `token_hash` співпадає. У разі будь-якого порушення повертається `401 Unauthorized` без розкриття причини.

Endpoint `POST /auth/logout` приймає refresh-токен, декодує його, знаходить запис у allowlist і встановлює `revoked=True`. Спроба використання після logout — `401`.

Зміна пароля (`PUT /auth/password`) і скидання пароля (`POST /auth/reset-password`) додатково викликають `revoke_all_for_user(user_id)`, що позначає revoked всі активні refresh-токени користувача. Це закриває критичний сценарій: викрадений refresh-токен втрачає чинність у момент, коли користувач помітив компрометацію та змінив пароль.

### 3.2.3 Захист від брутфорсу — блокування облікового запису

Класичний rate-limit на базі IP неефективний проти credential stuffing з residential-проксі мережі: атакуючий може робити запити з тисячі IP-адрес, обходячи per-IP обмеження. Для розв'язання реалізовано per-account блокування у `services/account_lockout.py`.

При кожній невдалій спробі входу інкрементується лічильник `failed_login_attempts`, фіксується час останньої невдачі. Якщо за п'ятнадцятихвилинне вікно накопичилося п'ять і більше невдач, обліковий запис блокується на п'ятнадцять хвилин шляхом встановлення `locked_until`. Усі наступні спроби входу — навіть з правильним паролем — повертають `423 Locked` із заголовком `Retry-After`, що містить кількість секунд, які залишилися до розблокування.

Успішний вхід очищає лічильник, що запобігає накопиченню старих невдач у користувача, який припустився однієї помилки. Поєднання per-IP rate-limit (через `slowapi`) з per-account lockout робить credential stuffing непрактичним, оскільки навіть з тисячі IP атакуючий зможе зробити лише п'ять спроб на конкретний обліковий запис за п'ятнадцять хвилин.

### 3.2.4 Хешування токенів скидання пароля

Токен скидання пароля видається через `POST /auth/forgot-password`, надсилається на електронну пошту користувача, дійсний п'ятнадцять хвилин. На відміну від класичної схеми «зберегти JWT у БД і порівняти», у розробленому сервісі в БД зберігається лише SHA-256 хеш токена (`password_reset_token_hash`). Це означає, що читання БД (через витік дампу або компрометацію адміністратора) не дає атакуючому жодного активного скидання — для відновлення пароля потрібен сирий токен, який існує лише у поштовій скриньці користувача.

При успішному скиданні пароля (`POST /auth/reset-password`) токен інвалідується через `$unset` поля `password_reset_token_hash` (одноразовий використання) та через `revoke_all_for_user` для всіх активних refresh-токенів.

Запит `POST /auth/forgot-password` повертає однакове повідомлення незалежно від того, існує користувач з вказаною адресою чи ні (захист від email-enumeration). Логіка пошуку та надсилання листа винесена у фоновий task через `BackgroundTasks` FastAPI, що робить час відгуку константним і запобігає side-channel атаці на основі вимірювання часу відповіді.

### 3.2.5 Дворівнева фільтрація доступу

Доступ до пошукової видачі регулюється функцією `vector_store_service.build_access_filter(user_role, user_faculty_id, user_group_id, user_year, user_level, target_doc_types)`, яка будує MongoDB-фільтр для pre-filter у Atlas Vector Search.

**Рівень 1 — рівень доступу документа (`access_level`).** Адміністратор бачить усі документи. Викладач — `public` + `restricted` + `faculty` свого факультету. Студент — `public` + `faculty` свого факультету. Документи рівня `restricted` недоступні студентам у жодному випадку.

**Рівень 2 — аудиторна фільтрація (тільки для студентів).** Кожен фрагмент документа має поля `target_group_ids`, `target_years`, `target_level`. Студент бачить фрагмент, якщо для кожного з цих полів виконується умова: значення поля містить ідентифікатор групи / рік / рівень студента, АБО поле повністю відсутнє у фрагмента (інтерпретується як «для всіх»). Комбінація `$or` з `$exists: false` реалізує цю логіку у єдиному pre-filter без додаткових JOIN.

Викладачі та адміністратори свідомо звільнені від аудиторної фільтрації — вони повинні бачити всі дані, оскільки відповідають на питання студентів різних груп.

## 3.3 Реалізація конвеєра пошуку

### 3.3.1 Модуль query_analyzer

Модуль `services/query_analyzer.py` реалізує етап аналізу запиту. Базовий клас `QueryAnalysis` — dataclass з полями `intent`, `confidence`, `is_personal`, `entities`, `reformulated_query`, `preferred_strategies`, `target_doc_types`. Властивості `needs_structured` та `has_temporal_constraint` використовуються downstream-кодом для прийняття рішень про вибір стратегій пошуку.

Функція `analyze_query(question, chat_history)` робить асинхронний виклик LLM з промптом, що містить опис очікуваної JSON-структури і кілька прикладів. Результат парситься через `_parse_json_payload` (виявляє JSON-блок у тексті відповіді через шаблон збалансованих дужок) і коерситься через `_coerce_analysis` — функцію, яка приводить будь-який варіант відповіді до валідного `QueryAnalysis` (невідомий intent → `general`, неприпустима стратегія відсіюється, confidence обмежується діапазоном [0,1]).

При недоступності LLM (timeout 12 секунд або помилка JSON-парсингу) спрацьовує fallback — `_keyword_fallback_analysis(question)`. Fallback виявляє день тижня через регулярні вирази для шести варіантів кожного дня (`понеділок|понедельник|monday`), класифікує запит як `schedule_lookup` чи `exam_dates` за наявності ключових слів зі словників, виявляє `is_personal=True` за займенниками першої особи. Fallback не вимагає мережі і виконується за мікросекунди, що гарантує відсутність повного збою чату навіть при недоступності LLM-сервісу.

### 3.3.2 Модуль retrieval_orchestrator

Модуль `services/retrieval_orchestrator.py` об'єднує результати трьох стратегій пошуку. Основна функція `retrieve(query, analysis, pre_filter, k)` запускає вибрані стратегії паралельно через `asyncio.gather` і потім фузить результати через `_reciprocal_rank_fusion`.

Стратегія `_run_vector` робить виклик `vector_store_service.similarity_search_with_score` з pre-filter, отримує top-K результатів, фільтрує за порогом косинусної схожості (`vector_score_threshold`, за замовчуванням 0,65). Експериментально встановлено, що без цього порогу шумові фрагменти (cosine 0,4-0,5) потрапляли у фузію та псували видачу — поріг застосовується саме на стороні стратегії, ще до RRF.

Стратегія `_run_hybrid` використовує `MongoDBAtlasHybridSearchRetriever` з LangChain, який внутрішньо комбінує vector-search і Atlas full-text search. RRF-ранг очищається з метаданих повернутих документів, оскільки він не сумісний з косинусом і не повинен використовуватися downstream.

Стратегія `_run_structured` (модуль `services/structured_retriever.py`) працює зовсім інакше — без векторів. Вона завантажує `documents.structured_records` для документів, доступних поточному користувачу, і фільтрує записи за двома критеріями: аудиторія (група/рік/рівень) і сутності з аналізу запиту (день, дисципліна, викладач). Кожен підходящий запис формує синтетичний фрагмент з оцінкою 0,95, що змушує RRF поставити його у топ.

Функція `_reciprocal_rank_fusion(rankings, k=60)` приймає список ранжованих списків документів і повертає новий рейтинг за формулою `score(d) = Σ 1/(k + rank_i(d))`. Параметр `k=60` обрано на основі емпіричних рекомендацій з оригінальної роботи [4]; зміна параметру у діапазоні 50..100 практично не впливає на якість, але виходи за межі діапазону погіршують поведінку.

### 3.3.3 Модуль reranker

Модуль `services/reranker.py` містить функцію `rerank(query, candidates, top_n)`, яка переоцінює top-50 кандидатів від оркестратора через LLM-крос-енкодер. Промпт містить запит і нумерований список кандидатів (кожен — заголовок джерела + перші 400 символів тексту); LLM повертає JSON `{"scores": [{"id": 0, "score": 8.5}, …]}`.

Раніше реалізована наївна оптимізація — пропуск перерейтингу при кількості кандидатів ≤ top_n — виявилася критичною помилкою. Якщо retrieval повернув 18 нерелевантних кандидатів, а system prompt вимагає вичерпності («перерахуй усі знайдені записи»), модель починала вигадувати неіснуючі факти, щоб "відповісти" на шум. У виправленій версії перерейтинг виконується завжди, коли кандидатів більше одного, з відсіюванням за порогом релевантності 1,0/10. Якщо всі кандидати нижче порогу, зберігається лише найвищий — захист від повної відсутності контексту, що збереже хоча б одне посилання на джерело.

### 3.3.4 Модуль chat (orchestration)

Модуль `api/v1/chat.py` склеює всі шари. Основна функція — `run_rag_chain(question, user_role, user_faculty_id, user_group_id, user_year, user_level, history)`:

```
analysis = await analyze_query(question, history)
pre_filter = build_access_filter(user_role, ...)
result = await retrieve(question, analysis, pre_filter, k=top_k_results)
ranked = await rerank(question, result.docs, top_n=15)

if max_score(ranked) < no_answer_score_threshold and not result.structured_records:
    return no_answer_response()

context = format_docs(ranked) + format_structured_context(result.structured_records)
answer = await chain.ainvoke({"context": context, "question": question, "history": history})

if _is_no_information_response(answer):
    answer = await self_correction(question, context, ranked)

return {"answer": answer, "sources": extract_sources(ranked, result.structured_records), ...}
```

No-answer guard зрівнює максимальну оцінку фрагмента з порогом `no_answer_score_threshold` (0,55). Якщо оцінка нижче і немає structured records — повертається відповідь «Не можу знайти відповідь...», що повністю запобігає галюцинації.

Self-correction loop запускається, якщо LLM з причинно-наслідкових причин відмовив у відповіді ("у наданому контексті немає інформації…") навіть при наявному контексті — це може бути викликано надто формальним стилем системного промпту. Виявлення робиться через простий пошук маркерних фраз (`_NO_INFO_MARKERS`), і у разі співпадіння виконується додатковий виклик з посиленим промптом.

## 3.4 Реалізація рольового доступу та адміністрування

Endpoint'и адміністрування захищені dependency `admin_only`, який є частковим застосуванням `require_role("admin")`. Будь-який запит від користувача з role ≠ admin отримає `403 Forbidden` ще до виконання тіла обробника.

Endpoint `PUT /admin/users/{id}/block` блокує іншого користувача (встановлює `is_active=False`). Реалізовано захист «не блокуй себе» — запит з `user_id == current_user._id` повертає `400 Bad Request`. Цей захист критичний: випадкове самоблокування єдиного адміністратора призводить до повної втрати контролю над системою. Аналогічний захист реалізовано для `PUT /admin/users/{id}/role` — адміністратор не може зменшити свою власну роль.

Усі привілейовані дії фіксуються через `audit_log.record_action(actor, action, resource_type, resource_id, metadata)`. Дії включають: `user.approved`, `user.rejected`, `user.blocked`, `user.unblocked`, `user.role_changed`, `user.password_changed`, `document.uploaded`, `document.deleted`, `chat.session_deleted`. Метадані містять корисні для розслідування поля (наприклад, попередня та нова роль для зміни ролі, кількість revoked-токенів для зміни пароля). TTL-індекс на `timestamp` з терміном 365 днів автоматично очищає застарілі записи.

## 3.5 Реалізація клієнтського додатку

### 3.5.1 Маршрутизація та аутентифікація

Frontend використовує Next.js App Router з двома групами маршрутів: `(auth)/` (login, register, forgot-password, reset-password) та `(dashboard)/` (chat, profile, admin/*). Серверний edge middleware (`src/middleware.ts`) виконує аутентифікаційну перевірку для кожного запиту до маршрутів `(dashboard)`: неавторизований користувач отримує redirect на `/login` ще до рендеру сторінки, авторизований користувач, що відкрив сторінку входу, перенаправляється на `/chat`. Для адміністративних маршрутів (`/admin/*`) додатково перевіряється роль; користувачі без admin-ролі отримують redirect на `/chat`.

Рання версія middleware була збережена у файлі з неправильним ім'ям (`proxy.ts` замість `middleware.ts`), що блокувало автозавантаження Next.js, і всі серверні перевірки виконувалися лише на стороні клієнта. Цю помилку було виявлено в ході аудиту коду та усунено перейменуванням файлу.

NextAuth.js керує життєвим циклом сесії на стороні клієнта: при вході зберігає JWT-пару у безпечному httpOnly-cookie, автоматично оновлює access-токен через refresh-токен за чверть години до закінчення, очищає сесію при logout.

### 3.5.2 Edge middleware і Content Security Policy

Окрім аутентифікації, middleware відповідає за встановлення Content Security Policy на основі унікального nonce, що генерується для кожного запиту. Конкретно, `script-src` дозволяє лише скрипти з `'self'` та з відповідним `nonce-XXXX` (Next.js автоматично проставляє цей атрибут на свої inline-скрипти при налаштованому nonce), плюс директива `'strict-dynamic'`. Це дозволяє повністю позбутися небезпечних `'unsafe-inline'` і `'unsafe-eval'` у script-src без поломки React-гідрації.

Додаткові директиви: `frame-ancestors 'none'` (запобігає clickjacking), `base-uri 'self'`, `object-src 'none'`, `form-action 'self'`, `upgrade-insecure-requests` (у production). Встановлюються також стандартні security-headers: `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Referrer-Policy: strict-origin-when-cross-origin`, `Permissions-Policy: camera=(), microphone=(), geolocation=()`, `Strict-Transport-Security`.

### 3.5.3 Інтерфейс чату з потоковим відображенням відповіді

Компонент `chat-message-list.tsx` рендерить історію повідомлень. Нове повідомлення додається в кінець списку відразу після надсилання запиту з порожнім контентом і станом «друкує»; контент заповнюється поступово в міру надходження подій SSE від сервера. Використовується стандартний браузерний `EventSource` з обробниками `message`, `error`, що дозволяє реалізувати плавне відображення без додаткових бібліотек.

Markdown-рендеринг виконується через `react-markdown` з обмеженим переліком дозволених елементів (`ALLOWED_ELEMENTS`) — це усуває векторну XSS-атаку через `<script>` чи `<iframe>` всередині відповіді LLM. Окремо реалізовано фільтр `isSafeExternalHref`, який пропускає лише `http(s)://` та `mailto:` URL; усі інші схеми (`javascript:`, `data:`, `vbscript:`, `file:`, протокол-відносні `//`) не рендеряться як активні посилання.

Цитати у відповіді у форматі `[N]` автоматично перетворюються на якоря `#src-N`, що дозволяє натиснути на цитату і перейти до відповідного джерела у списку нижче.

### 3.5.4 Адміністративні сторінки

Сторінки `/admin/*` використовують спільний layout з sidebar-навігацією. Кожна сторінка має власний хук для роботи з даними:

- `useAdminUsers` — пагінований список користувачів з фільтрацією за статусом, дії блокування / зміни ролі;
- `useAdminDocuments` — список документів з можливістю перегляду тексту та видалення;
- `useAdminAnalytics` — агрегати запитів і входів за днями для дашборду;
- `usePendingTeachers` — заявки викладачів, що очікують підтвердження.

Завантаження документа реалізовано через окремий `DocumentUpload` компонент з drag-and-drop, валідацією формату/розміру на стороні клієнта (Zod-схема), вибором рівня доступу та цільової аудиторії (групи, роки, рівень навчання) через каскадні селекти.

### 3.5.5 Інтернаціоналізація

Усі видимі рядки інтерфейсу виносяться у файли локалізації `messages/en.json` і `messages/uk.json` (за схемою `domain.subdomain.key`). Доступ через хук `useTranslations(namespace)`. Активна локаль визначається за HTTP-заголовком `Accept-Language` з можливістю явного перемикання користувачем; вибір зберігається у cookie `NEXT_LOCALE`.

Дати форматуються через `toLocaleDateString(locale)`, де `locale` отримується з `useLocale()`, що забезпечує коректне відображення місяців та днів тижня обома мовами без жорстко прописаних рядків.

## 3.6 Особливості реалізації, що забезпечують надійність

**Fail-fast при старті у production-середовищі.** Lifespan-обробник FastAPI підіймає виняток, якщо MongoDB недоступна або модель вбудовувань не вдалося завантажити. Це гарантує, що orchestrator (Kubernetes, Docker Compose) перезапустить контейнер замість того, щоб тримати веб-сервіс, який повертає 500 на кожен запит.

**Warm-up моделі вбудовувань при старті.** FastEmbed-модель multilingual-e5-large важить ~2,2 ГБ і потребує кількох секунд для завантаження в пам'ять. Lazy-завантаження при першому запиті означало б, що перший користувач після старту контейнера чекає 5-10 секунд (та з високою ймовірністю отримує timeout). Завантаження винесено в startup-хук через `asyncio.to_thread`, що звільняє event loop і дозволяє іншим задачам startup виконуватися паралельно.

**Атомарні операції видалення документа.** Endpoint `DELETE /documents/{id}` спочатку видаляє фрагменти з векторного індексу (за `document_id`), потім видаляє запис з `documents`. Об'єднання на `document_id` (а не на `source_file`) усуває критичну вразливість: завантаження файлу з тим самим іменем, що у документа жертви, з подальшим видаленням свого документа призводило б до видалення фрагментів жертви. Поточна реалізація захищена від такої атаки на рівні архітектури.

**Per-call file_content для парсера розкладу.** Раніше реалізована схема, при якій схеме `_ScheduleExtractor` зберігав байти PDF в атрибуті інстанса (`set_file_content`), створювала race condition — два паралельні завантаження могли підставити байти один одного. Виправлено передачею байтів як аргументу методу `extract(file_content=...)`; інстанс екстрактора більше не має мутабельного стану.

**Захист OOXML від zip-bomb.** Парсери DOCX/XLSX перед розпакуванням проходять через `_check_zip_bomb`, що читає central directory ZIP-архіву і перевіряє: жоден компонент не оголошує розміру понад 50 МБ, сумарний оголошений розмір — не понад 100 МБ. Перевірка робиться без розпакування, тому сама по собі не може бути використана для DoS.

**Обмеження частоти всіх endpoint'ів.** Через декоратор `@limiter.limit(...)` зі slowapi накладено обмеження на всі endpoint'и: 5/min для реєстрації, 10/min для входу та оновлення токена, 3/min для запиту скидання пароля, 30/min для більшості адміністративних дій, 20/min для пошуку. Обмеження працюють per-IP; для production-середовища у налаштуваннях змінено storage_uri на Redis, щоб обмеження зберігалися спільно між кількома worker-процесами.

# 4 ФУНКЦІОНУВАННЯ ТА ЕКСПЕРИМЕНТАЛЬНА АПРОБАЦІЯ

## 4.1 Функціонування з точки зору користувача

### 4.1.1 Студент

Типовий сценарій використання студентом починається з реєстрації. На сторінці `/register` студент вводить адресу електронної пошти (за регулярним виразом валідності), пароль (мінімум вісім символів, одна велика, одна мала, одна цифра), повне ім'я, та обирає факультет, групу, рік і рівень навчання з каскадних селектів (доступні значення завантажуються з довідника без аутентифікації — endpoint `/dictionaries/faculties/public`). Після подання форми обліковий запис створюється у статусі `is_approved=False`. Адміністратор отримує сповіщення на dashboard про нового користувача, що очікує підтвердження.

Після підтвердження адміністратором студент входить на сторінці `/login`, отримує JWT-пару у безпечних cookie і перенаправляється на сторінку чату `/chat`. Інтерфейс чату складається з sidebar з історією сесій (зліва), області повідомлень (центр) і поля вводу (знизу).

При першому запиті у новій сесії генерується UUID-ідентифікатор сесії, який зберігається у URL (`/chat/[sessionId]`). Питання надсилається на endpoint `POST /api/v1/chat/ask/stream`, відкривається SSE-з'єднання, і відповідь поступово відображається на екрані. Після завершення відповіді під нею з'являється список джерел (компонент `SourceCitation`), згрупованих за документом — натискання на джерело відкриває діалог із попереднім переглядом тексту фрагмента, з можливістю відкрити повний документ.

У відповіді цитати у форматі `[N]` рендеряться як активні посилання, що при натисканні розгортають відповідну картку джерела і прокручують до неї. Це дозволяє користувачу швидко перевірити обґрунтування будь-якого твердження у відповіді.

Кнопки оцінки відповіді (👍 / 👎) поряд з кожним повідомленням асистента зберігають зворотний зв'язок у колекцію `feedback`. Адміністратори можуть переглядати агреговану статистику зворотного зв'язку у адміністративному дашборді.

Студент може бачити лише ті документи, які доступні за його роллю та аудиторією. Якщо студент СА-4 запитає «який у мене розклад на четвер?», система знаходить відповідні фрагменти лише з його розкладу; розклад іншої групи не може бути включено у відповідь, навіть якщо в ньому є слова «розклад» і «четвер».

### 4.1.2 Викладач

Реєстрація викладача аналогічна до реєстрації студента, з тією різницею, що замість групи / року / рівня викладач вказує кафедру та посаду. Підтвердження адміністратором обов'язкове.

Після входу викладач має доступ до того самого інтерфейсу чату, з відмінністю в обсязі видимих документів: викладач бачить документи рівнів `public` і `restricted`, а також `faculty` свого факультету. Аудиторна фільтрація для викладачів вимкнена — викладач часто допомагає студентам різних груп і повинен мати повний огляд документації.

Окрім чату, викладачу доступна сторінка завантаження документів. Інтерфейс завантаження дозволяє: обрати файл (drag-and-drop або діалог вибору), задати рівень доступу, цільові групи (список ідентифікаторів через інтерактивний multi-select), цільові роки (multi-checkbox 1..6) і рівень навчання. Після завантаження статус прогресу відображається у тулбарі; завершення супроводжується повідомленням про кількість створених фрагментів.

Викладач може видалити лише ті документи, які завантажив сам — endpoint `DELETE /documents/{id}` перевіряє відповідність `uploaded_by_id` поточному користувачу.

### 4.1.3 Адміністратор

Адміністративний дашборд (`/admin`) показує agregовану статистику: загальну кількість користувачів, кількість заявок викладачів, що очікують підтвердження, статистику документів (за типом і рівнем доступу), стан системи (доступність MongoDB, активність векторного сховища). Адміністратор має доступ до сторінок:

- `/admin/users` — повний список користувачів з фільтрами за статусом, можливість підтвердження, відхилення, блокування, зміни ролі;
- `/admin/documents` — повний список усіх документів незалежно від доступу, з можливістю попереднього перегляду та видалення;
- `/admin/dictionaries` — управління довідниками факультетів та груп;
- `/admin/analytics` — графіки запитів, входів, активних користувачів за обраний період (1, 7, 30, 90 днів).

Усі привілейовані дії адміністратора фіксуються у журналі аудиту з прив'язкою до actor_id та resource_id.

## 4.2 Тестова інфраструктура

Тестове покриття реалізовано на основі pytest з підтримкою asyncio. Загальний обсяг — 256 автоматизованих тестів (1 пропускається при відсутності змінної середовища `EVAL_REGRESSION=1`, 1 явно відмічений як expected-fail для майбутнього сценарію). Тести розділено на чотири шари:

**Юніт-тести** (`tests/unit/`, 75 тестів) — модулі без зовнішніх залежностей: хешування паролів, видача й декодування JWT, безпека JWT (alg none, чужий ключ, чужий algoritm, відсутність обов'язкових claims, чужий issuer/audience), Pydantic-моделі, обмеження частоти, RBAC-залежності, prompt safety, error handler, audit log.

**Тести шару сервісів** (`tests/services/`, 73 тести) — модулі бізнес-логіки з мокованими залежностями: query analyzer (LLM mocked, fallback паттерни), structured retriever (audience і entity matching), vector store (filter builder для всіх ролей × всіх audience), atlas client, RAG pipeline orchestration.

**Інтеграційні тести API** (`tests/api/`, 52 тести) — endpoint'и з тестовим клієнтом: реєстрація, вхід, оновлення токена, скидання пароля, профіль, адміністрування, завантаження документів (включно з security-тестами на обхід обмежень: оверсайз, MIME-spoof, path-traversal, ZIP-bomb), захист від IDOR (студент B не може прочитати сесію студента A, викладач не може видалити чужий документ, студент іншого факультету не може переглянути документ), self-protection адміністратора (не може заблокувати або деградувати свій обліковий запис).

**End-to-end тести** (`tests/test_e2e.py`, 27 тестів) — повний цикл реєстрація → вхід → завантаження → пошук, з мокованими LLM- та векторним пошуком, але реальним FastAPI-додатком.

**Eval-харнес** (`tests/eval/`, 24 тести) — окремі юніт-тести метрик пошуку (Recall@k, Precision@k, MRR, nDCG@k), валідація схеми датасету, регресійний тест якості пошуку (опціональний, потребує живої MongoDB Atlas з завантаженим корпусом).

Час прогонки повного тестового набору на стандартному ноутбуку — близько 30 секунд. Сумарне покриття коду пакетом `pytest-cov` становить понад 80 % для критичних модулів безпеки та бізнес-логіки.

### 4.3.3 Метрики оцінювання

Якість пошуку оцінено за чотирма стандартними метриками інформаційного пошуку:

- **Recall@k** = (кількість релевантних серед top-k) / (загальна кількість релевантних). Показує, наскільки ретрівер «дотягнувся» до всіх релевантних фрагментів.
- **Precision@k** = (кількість релевантних серед top-k) / k. Показує частку релевантних серед повернутих.
- **MRR (Mean Reciprocal Rank)** = середнє за всіма питаннями значення `1/rank`, де `rank` — позиція першого релевантного фрагмента у видачі.
- **nDCG@k (Normalized Discounted Cumulative Gain)** = відношення фактичного DCG до ідеального; враховує позицію кожного релевантного фрагмента (релевантний на позиції 1 кращий за релевантного на позиції 5).

Усі метрики усереднюються по дванадцяти тестових питаннях. Параметр `k` фіксовано на значенні 5, що відповідає середній кількості фрагментів, які користувач перегляне у списку джерел.

### 4.3.5 Аналіз отриманих результатів

Аналіз метрик у таблиці 4.2 виявляє чітку структуру: запити, що містять конкретну назву дисципліни (питання 1-6), демонструють високу якість пошуку (Recall@5 в межах 0,25-1,00, MRR=1,0 для кожного), тоді як запити, побудовані лише на коді спеціальності або типі документа (питання 7-11), мають близькі до нуля метрики.

Це відповідає теоретичним очікуванням від моделі семантичних вбудовувань: текстова відстань між запитом «розклад магістрів 172 ТКРТ» і фрагментом «type: class; group: ТКРТ; subject: Антени та поширення радіохвиль» невелика, оскільки спільна лексика «розклад / клас» зустрічається у будь-якому розкладному фрагменті. Розрізнення між «розклад ТКРТ» і «розклад ПФНМ» вимагає, щоб embedding специфічного коду спеціальності переважив над embeddings'ами загальних термінів — на що моделі загального призначення не натреновані. Тому такі запити в реальній системі мають оброблятися через шар структурованого фільтру (за метаданими source_file або doc_type), а не через чисто векторний пошук. Цей висновок підтверджує доцільність обраної мультистратегічної архітектури з паралельним структурованим і векторним пошуком.

Pure-vector ретрівер (без structured-retrieval компоненти, який використовується у тестовому скрипті) не може досягти оптимальних результатів на запитах за спеціальністю — повноцінна система чату включає структурований шар, який обробляє ці запити точним фільтром за `source_file` чи `doc_type`. Тестовий набір свідомо покриває повний спектр запитів, включаючи «складні» для ретрівера, щоб мати реалістичну оцінку нижньої межі продуктивності.

Найкращим питанням є «Розробка інтерфейсів користувача» (Recall@5=1,00, MRR=1,00, nDCG=1,00) — повна відповідність, що демонструє ефективність обраної моделі вбудовувань для україномовних дисциплін з характерними технічними термінами. Найслабшими — питання за загальною спеціальністю, що очікувано згідно з аналізом вище.

### 4.3.6 Регресійний контроль

Метрики Таблиці 4.2 збережено у файлі `tests/eval/baseline.json` як еталонні значення. На їх основі автоматизований тест `test_retrieval_meets_baseline` виконує повний прогон оцінювання при кожному запуску (за умови встановленої змінної середовища `EVAL_REGRESSION=1`) і провалюється, якщо будь-яка метрика впаде нижче 90 % від еталонного значення. Поріг у 90 % обрано як компроміс між чутливістю до регресії та толерантністю до non-determinism у нейронних моделях вбудовувань (різні запуски можуть давати трохи різні вектори через різницю у ядрах ONNX Runtime).

Така автоматизація гарантує, що зміна моделі вбудовувань, конфігурації pre-filter, алгоритму RRF чи будь-якого іншого компонента ретрівера буде помічена ще на етапі тестування, до розгортання у production.

Час обробки одного документа залежить від типу. Для PDF-розкладу з 90 заняттями на 3 сторінках:

- Контекстуалізація (LLM) — 0,3 с;

- **Усього: ~17 с.**

- Парсинг PDF — 5 с;
- Класифікація + контекстуалізація — 0,8 с;
- Розбиття за заголовками — 0,2 с;
- Embedding ~150 фрагментів — 2,5 с;
- Запис — 0,8 с;
- **Усього: ~9 с.**

Обидва значення відповідають типовим очікуванням користувача при адміністративній операції завантаження документа.

### 4.4.3 Споживання пам'яті

Завантаження моделі multilingual-e5-large у пам'ять процесу backend займає близько 2,3 ГБ. Кеш Pydantic-схем, MongoDB driver і FastAPI-роутер додають ще ~150 МБ. Стійке споживання пам'яті при обслуговуванні 100 одночасних з'єднань — близько 2,7 ГБ, що дозволяє розгортати backend на віртуальній машині з 4 ГБ RAM з достатнім запасом для пікових навантажень.

Frontend після `next build` має сумарний розмір клієнтських ресурсів близько 1,2 МБ (gzipped: ~350 КБ), що відповідає рекомендаціям сучасних практик веб-розробки.

Кількісна оцінка отриманих результатів демонструє відповідність розробленого програмного засобу сформульованим вимогам:

- На експериментальному корпусі з 7 документів і 623 фрагментів отримано середнє значення Recall@5 = 0,31, MRR = 0,54, nDCG@5 = 0,35, що вище мінімальних порогів, визначених у нефункціональних вимогах (Recall@5 ≥ 0,28, MRR ≥ 0,49, nDCG@5 ≥ 0,31).

- Реалізовано 256 автоматизованих тестів, що покривають критичні шляхи безпеки (JWT, RBAC, IDOR, lockout), бізнес-логіки (RAG-конвеєр, парсер розкладів) і якості пошуку (regression-gate з прив'язкою до контрольних значень).
- Frontend збирається без помилок типізації та linting'у, gzipped-розмір клієнтських ресурсів — 350 КБ.

**Ступінь впровадження та можливі галузі використання.** Розроблений сервіс готовий до пілотного впровадження на рівні факультету або кафедри. Архітектура передбачає горизонтальне масштабування, що дозволяє довести його до промислової експлуатації на рівні цілого університету. Можливі галузі застосування: довідкові сервіси для здобувачів і викладачів ВНЗ; внутрішні бази знань кафедр з авторегульованим доступом; платформи дистанційного навчання з пошуком за матеріалами курсу; адміністративні системи з пошуком у нормативно-правовій документації.

**Інформація щодо створення нової апаратури, приладів тощо.** Робота не передбачає розробки нової апаратури — це програмний засіб, що працює на стандартному серверному обладнанні (віртуальна машина з 4 ГБ RAM, MongoDB Atlas як зовнішній сервіс).

**Наукова, науково-технічна, соціально-економічна значущість роботи.** Робота демонструє практичне застосування сучасних архітектур RAG і векторного семантичного пошуку до реального україномовного академічного корпусу. Запропонований гібридний підхід (детермінований парсер табличних структур + LLM-уточнення) суттєво підвищує точність атрибуції даних з типового для університету формату навчального розкладу порівняно з прямим використанням LLM-видобування. Соціально-економічна значущість полягає у скороченні часу пошуку інформації для здобувачів і у зменшенні навантаження на адміністративний персонал факультетів.

**Доцільність продовження робіт за відповідною тематикою.** Подальший розвиток системи може бути направлений у кілька напрямків:

- Розширення тестового корпусу та exhaustive-eval з 50-100 питаннями для більш статистично значимої оцінки якості пошуку.

- Додавання модуля проактивних сповіщень (зміни розкладу, нові накази) з push-нотифікаціями.
- Інтеграція з системою електронного журналу університету для надання відповідей на питання про оцінки та академічну успішність.
- Підтримка голосового вводу запиту через Web Speech API для покращення доступності.

## Реферат (зразок)

РЕФЕРАТ

**Ключові слова:** RAG, RETRIEVAL-AUGMENTED GENERATION, ВЕЛИКА МОВНА МОДЕЛЬ, СЕМАНТИЧНИЙ ПОШУК, ВЕКТОРНІ EMBEDDING, MONGODB ATLAS VECTOR SEARCH, FASTAPI, LANGCHAIN, MULTILINGUAL E5, РОЛЬОВИЙ ДОСТУП, ВИДОБУВАННЯ СТРУКТУРОВАНИХ ДАНИХ, КОНТЕКСТУАЛЬНИЙ ПОШУК, RECIPROCAL RANK FUSION, NEXT.JS, JWT.

**Мета роботи** — розробити веб-систему семантичного пошуку відповідей в університетській документації на основі архітектури Retrieval-Augmented Generation з підтримкою рольового доступу, аутентифікації та видобування структурованих даних з графіків занять і протоколів сесії.

За результатами аналізу типових інформаційних потреб студентів, викладачів і адміністрації факультету розроблено повнофункціональну веб-систему, яка складається з backend-сервісу на FastAPI з векторним сховищем MongoDB Atlas Vector Search, frontend-додатку на Next.js, та конвеєра обробки документів. Реалізовано детермінований парсер навчальних розкладів, який точно атрибутує заняття до груп через структурний аналіз таблиць PDF замість послідовності токенів LLM. Впроваджено триетапну архітектуру пошуку: аналіз запиту LLM → мультистратегічне отримання → перерейтинг.

**Методи:** ретрівл-аугментована генерація, контекстуальний пошук Anthropic, гібридний пошук з фузією рангів (Reciprocal Rank Fusion), векторні embedding'и multilingual-e5-large, LLM cross-encoder reranking, рольова модель доступу (RBAC).

**Інструменти:** Python 3.11, FastAPI, LangChain, MongoDB Atlas Vector Search, FastEmbed, Deepseek API, Next.js 15, React 19, TypeScript, Tailwind CSS, shadcn/ui, NextAuth.js, Pydantic v2, pdfplumber, pytest. Розроблений комплекс може застосовуватися як основа для впровадження інтелектуальних довідкових сервісів у вищих навчальних закладах України.

## Конфігурація MongoDB Atlas Vector Search index

Конфігурація вектор-search індексу `vector_index` для колекції `document_chunks`:

```json
{
  "name": "vector_index",
  "type": "vectorSearch",
  "fields": [
    {
      "type": "vector",
      "path": "embedding",
      "numDimensions": 1024,
      "similarity": "cosine"
    },
    { "type": "filter", "path": "access_level" },
    { "type": "filter", "path": "faculty_id" },
    { "type": "filter", "path": "target_group_ids" },
    { "type": "filter", "path": "target_years" },
    { "type": "filter", "path": "target_level" },
    { "type": "filter", "path": "doc_type" }
  ]
}
```

Конфігурація full-text-search індексу `text_index` для гібридного пошуку:

```json
{
  "name": "text_index",
  "mappings": {
    "dynamic": false,
    "fields": {
      "text": { "type": "string", "analyzer": "lucene.standard" }
    }
  }
}
```
